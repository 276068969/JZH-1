# 注册失败时 profile 误报「资料完整」问题修复记录

## 一、问题现象

用户在注册表单提交后，若因**字段格式错误**（如身份证校验位错误、信用代码格式不符）或**唯一性冲突**（用户名已占用、邮箱已注册）导致注册失败，后端返回的 `profile.complete` 仍会被判定为 `true`，`profile.nextStepHint` 仍会显示「资料完整，可直接浏览车辆并下单」或「资料完整，可前往企业用车通道申请批量租车服务」。

这种错误信号会通过前端的 Alert 组件（成功样式）展示给用户，与 `fieldErrors` 里的校验失败信息并存，严重误导用户：
- 用户会看到「资料完整」的绿色/蓝绿色提示，与下方红色标红的字段错误冲突
- 下一步行为建议错误，用户可能被引导到其他页面而不是修正当前表单

## 二、根因分析

### 2.1 后端根因

`UserServiceImpl.buildProfileInfo(userType, req, forceComplete)` 仅基于「必填字段是否有值」判断完整度：

```java
List<String> missing = new ArrayList<>();
for (String f : required) {
    Object v = getFieldValue(req, f);
    if (v == null || !StringUtils.hasText(v.toString())) missing.add(f);
}
boolean complete = forceComplete || missing.isEmpty();
```

这会遗漏两种「字段已填写但内容无效」的场景：

| 场景 | missing 列表 | complete 判定 | 实际情况 |
|---|---|---|---|
| 身份证号填了 18 位但校验位错 | 空 | `true`（误） | 内容无效，需要修正 |
| 邮箱格式错、手机号非 1 开头 | 空 | `true`（误） | 内容无效，需要修正 |
| 用户名已被占用（唯一性校验） | 空 | `true`（误） | 需更换用户名 |
| 信用代码重复，已被注册 | 空 | `true`（误） | 需更换或联系管理员 |

同时 `nextStepHint` 没有区分「有错误」「字段缺失」「完全没问题」三种场景，统一复用了「资料完整」的文案。

### 2.2 前端根因

前端 `Register.tsx` 中的 Alert 样式判断仅依赖 `serverErrors` 的长度：

```tsx
type={Object.keys(serverErrors).length ? 'warning' : 'success'}
```

虽然在大多数情况下 `fieldErrors → applyServerErrors → serverErrors` 链路是同步的，但存在以下风险：
1. 如果未来出现「有 errors 但 serverErrors 还未 flush」的时序问题，仍会误显 success
2. 更严重的是：即使显示样式是 warning，`profileHint` 里的文案仍可能是「资料完整…」，后端不修复时前端根本无法兜底

## 三、修复方案

### 3.1 后端修复（核心修复）

**文件**：`backend/src/main/java/com/carrental/service/impl/UserServiceImpl.java`

#### 变更点 1：`buildProfileInfo` 签名扩展

新增第 4 个参数 `List<RegisterResponseVO.FieldError> errors`，将 errors 纳入完整度决策。

#### 变更点 2：complete 判定新公式

```java
boolean hasAnyError = errors != null && !errors.isEmpty();
boolean complete = !hasAnyError && (forceComplete || (missing.isEmpty() && invalidFields.isEmpty()));
```

即：只要存在任意字段错误 → `complete = false`，从根本上阻断误判。

同时将 errors 中涉及的字段也合并进 `missing`，便于前端统计缺失项（即使字段本身非空）。

#### 变更点 3：nextStepHint 三分法分支

```
if hasAnyError:
    「企业/实名资质存在 N 项问题，请修正标红字段后重新提交（身份证号、手机号…）」
else if missing:
    「请补充 XX 后再提交申请：缺少 公司全称、法人身份证号」
else:
    原 success 文案：资料完整，可前往… / 可直接…
```

#### 变更点 4：辅助基础设施

- 新增 `FIELD_LABELS` 静态映射，字段英文 key → 中文标签
- 新增 `joinFieldLabels(List<String>)`，把 `[idCard, phone]` 转成「身份证号、手机号」
- 成功注册分支的 `buildProfileInfo` 调用也同步更新为新签名，传空列表 errors（语义清晰）

### 3.2 前端修复（双层兜底）

**文件**：`frontend/src/pages/Register.tsx`

#### 兜底 1：独立的提交状态 `submitStatus`

新增状态机 `'idle' | 'success' | 'error'`，在 `onFinish` 中明确设置：
- 成功分支 → `'success'`（配合 message.success + setTimeout 跳转）
- 失败分支（业务 422 / catch 异常） → `'error'`
- 切换 userType / 重新提交前 → `'idle'`

Alert 样式判断改为：

```tsx
type={
  submitStatus === 'error' || Object.keys(serverErrors).length > 0
    ? 'warning'
    : submitStatus === 'success' ? 'success' : 'info'
}
```

即 `submitStatus === 'error'` 优先匹配 warning，即使极端情况下 serverErrors 为空也不会误显 success。

#### 兜底 2：错误场景下 `profileHint` 文案清洗

新增 `sanitizeErrorProfileHint(raw, userType, errorCount)`：
- 如果 `errorCount > 0` 且 `raw` 包含「资料完整、可直接、可前往、可浏览、下单、申请」等误导性关键词 → **直接覆盖**为安全文案：「企业资质 / 实名信息存在 N 项问题，请修正页面中标红的字段后重新提交」
- 只有通过关键词检查的后端原始文案才被保留

#### 兜底 3：切换账户类型重置状态

`useEffect` 切换 personal/enterprise 时，除了清空 serverErrors，还额外 `setProfileHint(null)` + `setSubmitStatus('idle')`，避免残留上一次提交的错误状态。

## 四、测试验证矩阵

| 用例 | 预期 profile.complete | 预期 nextStepHint 关键词 | 预期 Alert type |
|---|---|---|---|
| 所有字段为空，点提交 | `false` | 请补充 | warning |
| 格式正确 + 唯一，注册成功 | `true` | 资料完整，可… | success |
| 所有必填都填了 + 身份证校验位 X 写成 0 | `false` | 存在 N 项问题，请修正 | warning |
| 所有必填都填了 + 用户名已被 test 占用 | `false` | 存在 N 项问题，请修正 | warning |
| 邮箱格式写成 `abc@`（不合法） | `false` | 存在 N 项问题，请修正 | warning |
| 信用代码 17 位，少一位 | `false` | 存在 N 项问题，请修正 | warning |

## 五、受影响文件清单

- `backend/src/main/java/com/carrental/service/impl/UserServiceImpl.java`
  - `buildProfileInfo` 签名 + 核心逻辑
  - `buildValidationError` 调用签名
  - 注册成功分支调用签名
  - 新增 `FIELD_LABELS` 映射
  - 新增 `joinFieldLabels` 工具方法
  - 新增 `import java.util.HashMap` / `import java.util.Map`
- `frontend/src/pages/Register.tsx`
  - 新增 `submitStatus` 状态
  - 新增 `sanitizeErrorProfileHint` 函数
  - `onFinish` 两分支中新增状态设置和兜底调用
  - `useEffect` 重置逻辑扩展
  - Alert `type` 判断逻辑增强

## 六、后续建议（可选）

1. 若后续补全流程新增「资料编辑」接口，建议复用同一套 `buildProfileInfo` 逻辑，保证完整度判定在全系统中一致。
2. 可考虑把 `FIELD_LABELS`、`joinFieldLabels`、`sanitizeErrorProfileHint` 这类与 UI 文案相关的逻辑抽成共享 util，避免前后端维护两份标签映射。
