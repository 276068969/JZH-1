# 测试账号登录问题修复记录

## 修复概述

修复测试账号 `admin/admin123` 无法登录的问题，同时补充完善登录鉴权全链路的错误处理，确保测试账号始终可用。

---

## 问题根因分析

### 1. 直接原因：数据库密码哈希无效

`database/init.sql` 中预置的用户密码哈希是伪造/无效的 BCrypt 哈希值：

```
$2a$10$N.zmdr9k7uOCQb376NoUnuTJ8iAt6Z5EHsM8lE9lBOsl7iAt6Z5EH
```

该哈希格式看似 BCrypt（`$2a$10$` 前缀），但实际无法匹配明文密码 `admin123`，导致 `PasswordEncoder.matches()` 返回 false，登录失败。

### 2. 放大原因：前端错误处理不严谨

前端登录页最初的实现逻辑存在缺陷：

- 仅检查 HTTP 状态码 200，未校验业务 `code` 字段
- 只要 HTTP 请求成功，无论后端返回 `code: 200` 还是 `code: 500`，都显示"登录成功"
- 导致出现"假登录"现象：页面显示登录了，但刷新后 token 无效又掉回登录页

### 3. 系统原因：缺乏测试账号保障机制

没有应用启动时的账号校验机制，仅依赖 SQL 脚本的初始数据，一旦数据损坏或哈希错误就会导致测试账号不可用。

---

## 修复内容

### 1. 新增测试账号初始化器（核心修复）

**修改文件**：
- `backend/src/main/java/com/carrental/config/TestAccountInitializer.java`（新增）

实现 `CommandLineRunner`，在应用启动时自动执行：

| 场景 | 处理方式 |
|------|----------|
| 账号不存在 | 使用 `PasswordEncoder` 生成正确的 BCrypt 哈希，创建账号 |
| 账号存在但密码不匹配 | 重新用 `PasswordEncoder` 编码默认密码，更新数据库 |
| 账号存在且密码正确 | 跳过，不做任何修改 |

**测试账号清单**：

| 用户名 | 密码 | 用户类型 | 邮箱 | 手机号 |
|--------|------|----------|------|--------|
| admin | admin123 | 企业用户 (enterprise) | admin@carrental.com | 13800138000 |
| test | test123 | 个人用户 (personal) | test@carrental.com | 13900139000 |

**设计特点**：
- 使用 Spring Security 的 `PasswordEncoder` 生成哈希，确保与登录验证逻辑一致
- 密码不匹配时自动重置，保证测试账号始终可用
- 异常时安全跳过，不影响应用主流程启动
- 日志输出 `[Test-Account]` 前缀，便于排查

### 2. 修复前端登录错误处理

**修改文件**：
- `frontend/src/pages/Login.tsx`

修复前逻辑：
```typescript
const response = await axios.post('/api/auth/login', values)
const token = response.data?.token
if (token) {
  localStorage.setItem('token', token)
}
message.success('登录成功！')  // 错误：无论成功失败都显示成功
onLogin()
navigate('/')
```

修复后逻辑：
```typescript
const response = await axios.post('/api/auth/login', values)
const code = response.data?.code
const token = response.data?.token

if (code === 200 && token) {
  // 只有业务 code 为 200 且 token 存在才认为登录成功
  localStorage.setItem('token', token)
  message.success('登录成功！')
  onLogin()
  navigate('/')
} else {
  throw new Error(messageText || '登录失败')
}
```

### 5. 修复后端 JWT 密钥长度问题

**修改文件**：
- `backend/src/main/java/com/carrental/config/JwtUtil.java`
- `backend/src/main/resources/application.yml`

问题：HS512 算法要求密钥至少 64 字节，原默认密钥仅 45 字节，导致 JWT 生成失败。

修复：
- 新增 `getSigningKey()` 方法，自动检测密钥长度
- 不足 64 字节时自动填充到 64 字节
- 同步延长配置文件中的默认密钥长度

---

## 验证方法

### 1. 测试账号登录验证

1. 启动后端服务，查看控制台日志，应看到：
   ```
   [Test-Account] Creating test account: admin
   [Test-Account] Test account 'admin' created successfully.
   [Test-Account] Creating test account: test
   [Test-Account] Test account 'test' created successfully.
   [Test-Account] Test accounts verification completed.
   ```

2. 重启服务，第二次启动应看到：
   ```
   [Test-Account] Test account 'admin' is up to date.
   [Test-Account] Test account 'test' is up to date.
   ```

3. 前端使用以下账号登录，均应成功：
   - admin / admin123（企业用户）
   - test / test123（个人用户）

### 2. 错误场景验证

1. 输入错误密码，应显示"用户名或密码错误"，不跳转
2. 输入不存在的用户名，应显示"用户名或密码错误"，不跳转
3. 删除 localStorage 中的 token 后刷新，应自动跳转到登录页
4. 使用无效 token 访问需要登录的页面，应自动登出

### 3. 密码重置验证

1. 手动修改数据库中 admin 用户的密码哈希为错误值
2. 重启后端服务
3. 验证 admin/admin123 仍可正常登录（初始化器自动重置了密码）
